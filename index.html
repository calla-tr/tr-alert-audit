Key Takeaway

Chainlink covers 3 of 12 alert patterns (Modal, FormErrorMessage, Tooltip). The remaining 9 patterns are custom-built independently in each app, leading to divergent implementations of the same concepts. The biggest opportunities are adding a Toast, Banner, InlineAlert, and Drawer component to Chainlink — these 4 additions alone would cover the majority of the fragmentation.

Consolidation Recommendations

Suggestions for reducing fragmentation and improving consistency across platforms.

1. Add Toast/Notification to Chainlink

High Priority — Biggest Gap

Chainlink has zero toast/notification support, forcing 3 independent implementations: Web Pills, Desktop Notification (7 variants), Desktop BannerNotification, Mobile Notification, and a legacy Angular Toaster.

  • Action: Add a Toast component to @trainerroad/chainlink-dom with variants: default, success, error, info, warning
  • Action: Include a ToastProvider with queue management, auto-dismiss (5s default), sticky option, and action button support
  • Action: Match the existing Pills styling (pill shape, 54px height, grey3 bg) as the standard
  • Action: Remove the legacy Angular Toaster immediately
  • Impact: Replaces 5 custom implementations with 1 shared component

2. Add Banner Component to Chainlink

High Priority — Biggest Gap

Banners exist in 5+ unrelated patterns: Web App Banners (3 with priority system), Desktop BannerNotification, InternetBanner, UpdateNotification, and WahooResyncBanner. None come from Chainlink.

  • Action: Add a Banner component to Chainlink with variants: info, warning, error, success, promotional
  • Action: Support props: icon, message, action button, close button, position (top/inline), theme
  • Action: Port the web priority system to @trainerroad/chainlink-shared as a BannerManager
  • Action: Unify DownloadApps + PublicProfile banners (identical dark style) into single component with props
  • Impact: One banner pattern instead of 5+ implementations

3. Add InlineAlert Component to Chainlink

High Priority — Biggest Gap

There are 4+ inline alert components (EstimateTssBanner, AlertBox, ListViewBanner, WorkoutProgressionsBanner) each with different styling, padding (12px vs 16px), and border-radius (2px vs 4px). Chainlink provides nothing here.

  • Action: Add an InlineAlert component with variants: info, warning, error, success
  • Action: Standard props: icon, title, message, closable, actions (button slots)
  • Action: Use Chainlink's existing theme system (alertThemeModeMap pattern already exists in web)
  • Action: Standardize on 4px border-radius and 12px padding
  • Impact: Replaces 4+ custom components across web and desktop

4. Unify Modal Systems onto Chainlink

High Priority

Chainlink provides Modal + BasicModal, but apps still maintain 5 additional modal implementations: Legacy ModalService (web), Legacy Modal (desktop), SimpleModal, SidePanelModal, and FullscreenModal.

  • Action: Fully deprecate Legacy ModalService (web) and Legacy Modal (desktop) — migrate all usages to Chainlink BasicModal
  • Action: FullscreenModal already maps to Chainlink's fullscreen variant — remove the custom component
  • Action: Replace SimpleModal usages with Chainlink Modal
  • Action: Add a ConfirmationDialog preset to BasicModal with destructive/non-destructive styling
  • Impact: Reduces 6 modal systems to 1 (Chainlink) + SidePanelModal for panel context

5. Add Drawer/SidePanel to Chainlink

Medium Priority

Both web (Drawer system, z-9998) and desktop (SidePanel, 3 sizes) have independent side panel implementations. This is a common pattern that belongs in the design system.

  • Action: Add a Drawer component with sizes: small (160px), medium (320px), large (480px)
  • Action: Support: header/footer slots, backdrop dismiss, ESC close, RTL, animation (150ms slide-in)
  • Action: Include a DrawerConfirmation sub-component for the common confirm-in-drawer pattern
  • Impact: Replaces 2 independent implementations with 1 shared component

6. Add PageError + SaveMessage to Chainlink

Medium Priority

ErrorMessage, PageErrorMessage, SaveMessage, and Notification (green bar) all exist as custom components in both web and desktop — SaveMessage is literally duplicated across repos. Chainlink only covers field-level errors.

  • Action: Add a PageErrorMessage component (icon + message, themed for light/dark)
  • Action: Add a SaveMessage component (green checkmark + "Saved" text, with fade animation)
  • Action: Standardize error colors using Chainlink's existing error/errorDark tokens
  • Impact: Eliminates duplicated components across repos

7. Add Accessibility Live Region to Chainlink

Medium Priority

Web has AccessibilityAnnouncementService for screen reader alerts; desktop and mobile have nothing. This is a cross-cutting concern that should be standard.

  • Action: Add an AccessibilityAnnouncement provider to Chainlink with polite/assertive priorities
  • Action: Auto-integrate with Toast and Modal components (announce on show)
  • Action: All new Chainlink alert components should include proper aria-live attributes
  • Impact: Consistent accessibility across all 3 platforms

8. Align Animation Patterns via Chainlink Tokens

Low Priority

Animation durations vary: 150ms, 200ms, 300ms, 400ms, 1000ms with different easing. Chainlink defines some (200ms modal, 300ms tooltip) but apps don't follow them consistently. Three different animation libraries are in use (react-pose, react-spring, CSS).

  • Action: Export animation duration/easing tokens from @trainerroad/chainlink-shared
  • Action: Standard durations: 150ms (micro), 200ms (feedback), 300ms (overlay)
  • Action: Migrate from deprecated react-pose to CSS transitions

9. Mobile Parity

Low Priority

Mobile has only 2 notification variants (success/error) while desktop has 7. Mobile lacks inline alerts and status indicators entirely.

  • Action: Once Toast and InlineAlert exist in Chainlink, create @trainerroad/chainlink-native equivalents
  • Action: Add warning and info variants to mobile Notification at minimum
  • Action: Ensure shared modals (RedDay, AI FTP, Adapt) have consistent behavior across platforms
